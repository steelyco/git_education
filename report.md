## *Git* - это распределенная система управления версиями. Благодаря git можно создавать разные версии приложений.

### Почему стоит использовать *git*
- Позволяет хранить разные версии приложения 
- Быстрый переход между версиями приложения
- Откат к предыдущим версиям
- Улучшает процесс командной работы над одним проектом 
- Можно независимо разрабатывать новые функции в приложении

### Разница между *git* и *github*
1. *git* - это распределенная система управления версиями. Работа только локально.
2. *github* - это сервис хостинга git репозиториев. Позволяет удаленно работать с git.

### Как поменять метаданные *git*
1. *git config --global user.name* <..> - настройка имени 
2. *git config --global user.email* <..> - настройка имейла
3. Проверить настройки можно *git config --list*

### В git существуют несколько **областей**, где хранятся файлы
1. Работая директория - *working directory* - файлы, которые лежат в папке 
2. Индекс - *staging area* - файлы, которые изменены, но не сохранены
3. Репозиторий - *repository* - тут хранятся файлы *.git*, разные версии файлов

> [!tip] Working directory - Работая директория
> - Файлы и папки, которые лежат в проекте

> [!tip] Staging area - Индекс
> - Файлы и папки, которые вы *подготавливаете для сохранения изменений в репозитории* - коммита
> - Посмотреть состояние индекса можно командой **git statis**

> [!tip] Repository - Репозиторий
> - Файлы и папки, которые находятся в **.git**
> - Сохраняются разные версии одного и того же файла 

## Статусы отслеживания файлов 
1. **Untracked** - *Не отслеживаемый* - файлы, которые только добавили в *Рабочую директорию*
2. **Staged** - *Подготовленный* - файлы, которые уже добавлены в staging area
3. **Unmodified** - *Не модифицированный* - файлы, которые только добавили в *Репозиторий* и пока не исправляли
4. **Modified** - *Модифицированный* - файлы, в которые внесли исправления, временное состояние файла 

## Типы объектов в *git*
1. **Blob** - *Файл* - Объекты создаются для файлов, для каждой версии файлов создается отдельный *Blob*
2. **Tree** - *Папки* - Содержит ссылки на те файлы, которые находятся в папке
3. **Commit** - *Коммит* - Объект в *git*, который создается после сохранения текущей версии проекта
4. **Annotated Tag** - *Аннотированный тег* - благодаря этому тегу можно перемещаться в по названию тега в разные версии проекта. Указывает на уже существующий *commit*

## **Commit** - Коммит
- Это указатель или ссылка на определенное дерево, то есть на корневую папку проекта, определенной версии
- В каждом *коммите* *git* создает новые объекты тех *Tree* и *Blob* в *Репозитории* которые, лежали в корневой папке
- Все объекты будут иметь уникальный *sha1 хэш*
- В разных версиях или *коммитах* будут разные корневые папки
- Каждый следующих *коммит* ссылается на предыдущий *коммит*
  
## **HEAD** - указатель на *Ветку*
- Следующая надстройка над *коммитом* является **HEAD**
- Благодаря **HEAD** можно перемещаться по *веткам*
- **HEAD** является указателем на ветку 

## **Ветка** - ссылается на определенный *commit*
- Можно перемещаться по *коммитам*
- При использовании одном **ветки** **HEAD** перемещается по *коммитам*
- **Ветка** всегда ссылается на самый последний *коммит*

## **Слияние** веток - *merge commit*
- Слияние побочной ветки в основную 
- Могут возникать конфликты *merge*, если были изменены одинаковые файлы
- После слияния у основного *коммита* теперь два родительских *коммита*
- Для слияния веток нужно находится **в той ветке**, в которую идет слияние
> [!tip] **Пошаговый процесс слияния веток:**
. Создается ветка *new-feature* из ветки *main*
. Переход в *new-feature*; Изменения в этой ветке; Создать *коммит*
. Вернуться обратно в *main*
. Выполнить слияние *new-feature* ветки в *main*


## Удаленный репозиторий - *github*
- Все git объекты находятся на удаленном сервере
- Можно настроить связь между своим локальным репозиторием и удаленным репозиторием
- Можно обновлять изменения, которые происходят в проекте командой - **git pull**
- Можно загружать изменения, которые происходят в проекте командой - **git push**
- **Origin** - это имя удаленного репозитория по умолчанию

## Базовые команды *git*

1. **git init** - создание локального репозитория 
	Нужно писать в папке, которая станет репозиторием
	Создается скрытая папка .git 
2. **git status** - текущее состояние индекса git или репозитория
3. **git add** < *files* > - подготовка для сохранения в репозиторий перед *коммитом*, но она добавляет файлы в *staging area*
4. **git commit -m** "*massage*" - сохранение текущей версии, *коммита* (всех изменений) в репозиторий, **commit** - папка *objects* в *.git* 
5. **git checkout** < *commit hash* > - позволяет перейти к определенной версии проекта (*коммиту*), тк теперь *git* управляет проектом
6. **git checkout** < *branch name* > - позволяет перейти на определенную *ветку* проекта
7. **git checkout -b** < *branch name* > - создает и перемещается на ветку
8. **git log** - просмотр истории изменений (*коммитов*), показывает все коммиты были сделано до последнего. Благодаря этой команде можно увидеть хэш *коммита* и переместиться к нему
9. **git cat-file -t** < *хэш объекта* > - можно увидеть тип объекта
10. **git cat-file -p** < *хэш объекта* > - можно увидеть содержание объекта
11. **git branch** - показывает список веток
12. **git branch** < *branch name* > - создает новую ветку
13.  **git branch -m** < *branch name* > - переименовывает ветку
14. **git branch -d** < *branch name* > - удаляет ветку, нужно находится на другой
15. **git merge** < *feature branch name* > - слияние веток, находясь в **main** или в той, в которую идет слияние
16. **git clone** < *url* > - клонирование удаленного репозитория к себе на компьютер
17. **git pull** - загружает и применяет изменения с удаленной ветки в локальную
18. **git push** - загрузка изменений из локальной ветки в ветку удаленного репозитория 
19. **git remote add origin** < *url* > - подключение удаленного репозитория в локальный
20. **git push -u origin** < *branch* > - загружает изменения, находясь в ветке. нужно для того, чтобы связать локальную ветку с удаленной и создать между ними связь, делается при первой загрузке изменений
